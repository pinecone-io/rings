# OpenTelemetry Integration

## Overview

rings emits OpenTelemetry traces and spans to give deep insight into workflow execution pace, cost distribution, and phase behavior. OTel is opt-in and adds no overhead when disabled.

Spans represent the hierarchical structure of a rings run: one root span per run, one child span per cycle, one grandchild span per individual phase run. This maps naturally to dashboards in Grafana, Honeycomb, Jaeger, or any OTel-compatible backend.

## Enabling OTel

OTel is **off by default**. Enable it by setting:

```bash
RINGS_OTEL_ENABLED=true
OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4317
```

If `RINGS_OTEL_ENABLED` is unset or set to anything other than `true`, rings uses a no-op tracer and incurs zero overhead.

## Environment Variables

### rings-controlled

| Variable | Default | Description |
|----------|---------|-------------|
| `RINGS_OTEL_ENABLED` | `false` | Set to `true` to enable OTel export |
| `RINGS_OTEL_INCLUDE_TOKENS` | `false` | Whether to include cost and token counts as span attributes and metrics. Default is `false` because OTel backends are often shared across teams and cost data may be sensitive. Set to `true` to include cost and token data in spans and metrics. |
| `RINGS_OTEL_STRIP_PATHS` | `false` | Set to `true` to strip absolute file paths from span attributes (replaces with basename only). Useful when workflow paths contain sensitive information like usernames or project names you don't want in telemetry. |

### Standard OTel variables (passed to rings's SDK)

| Variable | Default | Description |
|----------|---------|-------------|
| `OTEL_EXPORTER_OTLP_ENDPOINT` | `http://localhost:4317` | OTLP collector endpoint (gRPC) |
| `OTEL_EXPORTER_OTLP_PROTOCOL` | `grpc` | Protocol: `grpc` or `http/protobuf` |
| `OTEL_SERVICE_NAME` | `rings` | Service name in traces |
| `OTEL_SERVICE_VERSION` | rings binary version | Service version |
| `OTEL_RESOURCE_ATTRIBUTES` | — | Additional resource attributes (e.g. `deployment.environment=prod`) |

### Claude Code's own OTel

rings passes all environment variables through to the `claude` subprocess unchanged. If Claude Code supports its own OTel integration, users should configure it via whatever environment variables Claude Code documents (e.g. `ANTHROPIC_*`). rings does not mediate, wrap, or interfere with Claude Code's telemetry. The two telemetry streams are independent.

## Trace Structure

One trace is created per `rings run` invocation. The trace ID is randomly generated by the OTel SDK (as required by the W3C trace context spec). The rings run ID is stored as a span attribute (`rings.run.id`), not derived into the trace ID.

```
Trace: rings.run  (trace_id = random, per W3C spec)
│
├─ Span: rings.run  [root]
│    duration: entire workflow execution
│    attributes: rings.run.id, context.dir, completion_signal,
│                max_cycles, phases (comma-separated), status (completed|canceled|max_cycles)
│
├─── Span: rings.cycle  [child of root, one per cycle]
│      duration: all phase runs in this cycle
│      attributes: cycle.number
│
└──────── Span: rings.phase.run  [child of cycle, one per individual run]
            duration: executor subprocess wall time
            attributes: (see below)
```

## Span Attributes: `rings.phase.run`

| Attribute | Type | Description |
|-----------|------|-------------|
| `rings.run.id` | string | Run ID (e.g. `run_20240315_143022_a1b2c3`) |
| `rings.workflow.file` | string | Absolute path to workflow TOML |
| `rings.cycle` | int | Cycle number (1-based) |
| `rings.phase.name` | string | Phase name (e.g. `builder`) |
| `rings.phase.iteration` | int | Iteration within cycle (1-based) |
| `rings.phase.total_iterations` | int | `runs_per_cycle` for this phase |
| `rings.run.global_number` | int | Global run number across all cycles |
| `rings.exit_code` | int | Exit code from `claude` subprocess |
| `rings.completion_signal_found` | bool | Whether completion signal was in output |
| `rings.files_changed` | int | Count of files changed in context_dir (if manifest enabled) |
| `rings.cost.usd` | double | Cost in USD (if `RINGS_OTEL_INCLUDE_TOKENS=true`) |
| `rings.tokens.input` | int | Input token count (if `RINGS_OTEL_INCLUDE_TOKENS=true`) |
| `rings.tokens.output` | int | Output token count (if `RINGS_OTEL_INCLUDE_TOKENS=true`) |

## Span Status

- `rings.phase.run` spans are marked `ERROR` if the `claude` subprocess exits non-zero.
- `rings.run` root span is marked `ERROR` if the workflow ended with a phase error.
- Cancellation sets the root span status to `OK` with attribute `rings.status = "canceled"`.
- Max cycles reached sets root span status to `OK` with `rings.status = "max_cycles"`.
- Completion signal detected: `rings.status = "completed"`.

## Metrics

In addition to traces, rings emits the following OTel metrics (when `RINGS_OTEL_ENABLED=true`):

| Metric | Type | Description |
|--------|------|-------------|
| `rings.runs.completed` | Counter | Incremented on each successful workflow completion |
| `rings.runs.canceled` | Counter | Incremented on each Ctrl+C cancellation |
| `rings.phase.run.duration` | Histogram | Wall time per phase run in milliseconds |
| `rings.cost.usd` | Histogram | Cost per phase run in USD |
| `rings.tokens.input` | Histogram | Input tokens per phase run |
| `rings.tokens.output` | Histogram | Output tokens per phase run |
| `rings.cycles.per_run` | Histogram | Number of cycles to completion |

All metrics carry `rings.phase.name` as an attribute for filtering. `rings.workflow.file` is intentionally excluded from metric attributes — it has unbounded cardinality (one value per unique workflow path) and would cause metric explosion in most backends.

### Metric Units

| Metric | Unit |
|--------|------|
| `rings.phase.run.duration` | `ms` (milliseconds) |
| `rings.cost.usd` | `{USD}` |
| `rings.tokens.input` | `{token}` |
| `rings.tokens.output` | `{token}` |
| `rings.cycles.per_run` | `{cycle}` |
| `rings.runs.completed` | `{run}` |
| `rings.runs.canceled` | `{run}` |

## OTel Initialization Failure

OTel initialization (connecting to the OTLP endpoint) happens at startup if `RINGS_OTEL_ENABLED=true`. If initialization fails (e.g., endpoint unreachable, misconfigured credentials):

- rings prints a warning to stderr: `Warning: OTel initialization failed: <reason>. Continuing without telemetry.`
- Execution proceeds normally using a no-op tracer.
- OTel init failures are never fatal. The user's workflow runs regardless.

This matches the intent of OTel being strictly opt-in and not affecting core functionality.

## Endpoint Security

rings does not validate the `OTEL_EXPORTER_OTLP_ENDPOINT` URL before using it. Users should ensure the endpoint is a trusted, TLS-secured collector. Sending telemetry to an untrusted endpoint may expose workflow metadata (run IDs, file paths, phase names). If `RINGS_OTEL_INCLUDE_TOKENS=true`, cost and token data are also sent. For shared OTel backends, consider setting `RINGS_OTEL_STRIP_PATHS=true` to redact filesystem paths from span attributes.

## Rust Implementation

Use feature-flagged OTel crates to keep binary size minimal when OTel is compiled out:

```toml
[dependencies]
opentelemetry = { version = "0.24", optional = true }
opentelemetry-otlp = { version = "0.17", features = ["grpc-tonic"], optional = true }
opentelemetry_sdk = { version = "0.24", features = ["rt-tokio"], optional = true }
opentelemetry-semantic-conventions = { version = "0.16", optional = true }

[features]
default = ["otel"]
otel = ["opentelemetry", "opentelemetry-otlp", "opentelemetry_sdk", "opentelemetry-semantic-conventions"]
```

The `src/telemetry.rs` module wraps all OTel calls. When `RINGS_OTEL_ENABLED` is false, `telemetry::init()` returns a no-op tracer that compiles to nothing at the call sites.

## Example: Dashboard Queries

With spans flowing to a Grafana Tempo backend (TraceQL) and metrics to Prometheus:

```traceql
# All phase runs that failed (Tempo TraceQL)
{ span.rings.phase.name = "builder" && status = error }

# All runs for a specific workflow (Tempo TraceQL)
{ span.rings.run.id =~ "run_20240315.*" }
```

```promql
# Average cost per phase over last 24h (Prometheus/Mimir)
avg by (rings_phase_name) (rings_cost_usd_sum / rings_cost_usd_count)

# P95 phase run duration per phase
histogram_quantile(0.95, rate(rings_phase_run_duration_ms_bucket[1h]))

# Total runs completed today
increase(rings_runs_completed_total[24h])
```
