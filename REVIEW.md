[2026-03-17 / Task 6: F-044 Quota Backoff] **Quota backoff retry loop and non-quota error handling.** Implemented a retry loop in the engine that wraps the executor spawn/wait logic. When a run exits with code 0, no retry logic is triggered. When a run exits non-zero, the error is classified (quota, auth, or unknown). If the classification is quota and `quota_backoff.should_retry()` returns true, the run log is written with a retry count (e.g., `007-retry-1.log`), then we wait using `interruptible_sleep` (already extracted in Task 1), and then we spawn a fresh executor handle to retry. The key design choice: **if a retry exits non-zero with a non-quota classification (auth or unknown), we exit immediately without consuming a retry attempt**. This prevents wasting retry quota on unrecoverable errors like invalid credentials. Per spec and plan D-4, this behavior is recorded. **On quota backoff exhaustion** (all retries consumed), we write `failure_reason = "quota"` to state.json and exit code 3. Timeout and cancellation are NOT retried; they break out of the retry loop immediately and proceed with the normal exit path. CLI arguments `--quota-backoff`, `--quota-backoff-delay`, and `--quota-backoff-max-retries` are applied as overrides to workflow config after TOML parsing, with the delay and max-retries flags gated to require `--quota-backoff` via clap's `requires` attribute. All 41 existing tests pass. Code quality gates: format ✓, clippy ✓, tests ✓.

[2026-03-17 / Task 4: F-058 Parent Run ID — Resume Behavior] **Architectural decision: Option A (spec-compliant).** When `rings resume <run_id>` is invoked, the engine creates a **new run directory** with a **new run_id**, writing the old run_id as `parent_run_id` in the new run's metadata. This matches the spec statement "creates a new run" in `specs/state/run-ancestry.md`. The new run starts with `ancestry_depth = 1` (or `depth(parent) + 1` if parent directory exists). This design ensures clear lineage and allows independent cost tracking per run, preventing circular dependencies where old and new runs share the same `run_dir`. Rationale: Option B (reuse same directory) would leave `parent_run_id = run_id` (self-reference), which violates spec intent and prevents proper lineage tracking. Option A is the only lineage-coherent choice. Implementation: `resume_inner` will generate a fresh `run_id` at the start, write the resumed run's ID as `parent_run_id`, and use a new `output_dir/<new-run-id>/` tree. The old run's `state.json`, `costs.jsonl`, and logs remain in `output_dir/<old-run-id>/` as a complete audit trail.

[2026-03-17 / Task 6: F-149 — Cost Spike Detection] Implemented cost spike detection to alert users when a single run's cost exceeds 5× the rolling average for that phase. After each successful cost entry is appended to costs.jsonl, `check_spike()` is called to detect spikes and emit `advisory_warning` JSONL events with the multiplier (e.g., "6.2× the rolling average"). None-cost entries are skipped: they don't add to the rolling window and don't count toward the 3-run minimum threshold for spike detection. Zero rolling averages are skipped to avoid false positives and division issues. Wrote 12 comprehensive unit tests covering boundary cases, rolling window capping, zero handling, integration with resume cost reconstruction, and multiple independent phase windows.
